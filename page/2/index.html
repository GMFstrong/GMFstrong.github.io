Linex的简单安全配置

关于阻止地址伪装：（基于iptable）
IP地址伪装常用于攻击者向服务器发送恶意数据包，它利用虚假地址发送构造数据包，从而发动DDos攻击。
首先，如果希望阻止IP伪装我们就要从以往尝试的虚假连接中得到一个IP地址或域名的列表，然后通过IPtable规则集限制除必须入站流量包以外的数据包。
步骤如下：
（1）设定允许数据包进入loopback接口的规则，从而满足本系统各个服务之间的正常通信。
iptables -A INPUT -i lo -j ACCEPT

（2）为本地系统发起的连接创建规则，该规则将允许所有外出流量包通过，包括远程服务器的响应，如web站点的访问：
iptables -A INPUT -m conntrack –ctstate RELATED,ESTABLISHED -j ACCEPT

（3）创建一个新链，用来存放阻止IP伪装的规则。
iptables -N blocked_ip
（4）将生成的新链插入到INPUT链中；
iptables -I INPUT 2 -j blocked_ip


（5）将需要阻止的虚假IP地址规则插入到blocked_ip中,通过该规则可以阻塞该地址的连接。
iptables -A blocked_ip -s 192.168.1.2 -j DROP

其次，通过/etc/host.conf文件阻止IP欺骗。打开该文件在文件中添加一下几行内容：
order hosts,bind
multi on
nospoof on
这里 “nospoof on”选项表示开启IP地址的比较操作，本地主机通过主机名查找对应的ip地址，并将查到的IP地址与数据包中的源IP比较，如果不一致，这说明
该IP为伪造ip，系统会发出虚假IP地址警告！

代码执行

关于代码执行漏洞：
定义:
由于应用程序本身过滤不严，用户可以通过请求把代码注入到应用中执行。直白一点，就是类似于SQL注入，可以把SQL语句注入到SQL服务中执行，
在PHP代码中执行漏洞则是可以把代码注入到应用中最终Webserver去执行。
这样的漏洞如果没有过滤，相当于一个web后门，相当严重。
成因；
该漏洞主要是由于eval(),assert(),preg_replace(),call_user_func(),call_user_func_array(),array_map()等函数过滤不严导致的。另外，PHP
的动态函数（$a($b)）也是目前出现比较多的。
挖掘经验：
eval(),assert()函数导致的代码执行漏洞大多数是因为载入模块或者缓存亦或者是对变量过滤不严导致的。例如：直接把外部可控的参数直接拼接到
模板里面，然后调用这两个函授去当成PHP代码执行。
preg_replace()函数的代码执行漏洞需要满足’/e‘参数存在，因为这个函数是处理字符串的，所以漏洞出现最多的是在对字符串处理上。比如在对URL
HTML标签以及文章内容过滤功能。
call_user_func()和call_user_func_array()函数的功能是调用函数，多用在框架里面动态调用函数，故而比较小的程序出现这种代码执行漏洞的机会
比较少。
array_map()函数的作用是调用函数并且除第一个参数外其他参数为数组，通常会写死第一个参数（调用的函数），类似的这类函数还有很多。除此引发
的代码执行漏洞，还有一种常见的动态函数的代码执行漏洞，例如：
$_GET($_POST[“XX”])基于这种写法变形出来的各种异形，经常被当做后门来使用。
PHP代码执行有多种利用方式，但目前最多的还是函数使用不当导致的，比如：eval(),assert(),preg_replace(),call_user_func(),call_user_func_array()
array_map()等。
eval()和assert():
这两个函数本来就是用来动态执行PHP代码的，所以他们的参数直接就是PHP代码。
preg_replace()：
preg_replace($pattern,$replacement,$subject)
该函数的作用是对字符串进行正则处理，但是在它的第一个参数$pattern存在e修饰时，$replacement的值会被当成PHP代码来执行。
例如：
<?php
preg_replace(“/[(.*)]/e”,’\1’,$_GET[‘str’]);
?>
调用函数过滤不严：
call_user_func()和call_user_func_array()的十个函数有调用其它函数的功能，其中一个参数作为要调用的函数名，如果传入的函数名可控，那就可以调用
意外的函数来执行我们想知道的代码，也就存在代码执行漏洞了。
例如：
<?php
$b = “phpinfo()”;
call_user_func($_GET[‘a’],$b);
?>
当我们传入X.php?a=assert时候，则调用assert函数，并且将phpinfo()作为参数传入。
动态函数执行：
由于PHP特性的原因，PHP函数可以由字符串直接拼接，这导致了PHP在安全控制上增大了难度，比如增加了漏洞数量并且增加了后门的查找难度。他们的初衷
和call_user_func()的初衷是一样的，都是为了更方便的调用函数，但是一旦过滤不严就会导致代码执行漏洞。
PHP动态函数写法为“变量（参数）”的形式。例如：
<?php
$_GET[‘a’]（$_GET[‘b’]）;
?>
即‘a’接收的为函数，‘b’接收的为参数。故而在‘a’参数为assert时，‘b’参数为phpinfo()时，为assert(phpinfo())。
在挖掘这种形式的漏洞时，需要寻找可控的动态函数名。
漏洞防范：
采用白名单过滤的方式，在满足业务的前提下，这是非常实用的方式，这里的白名单不是完全固定参数，而是结合正则表达式来进行白名单限制，例如如果我
们事先知道参数位数字型的就可瞒住业务要求，那么在正则里面只需要用/d+来限制即可！！！这样相对更加安全。

文件读取（下载，上传）

关于文件读取：
部分程序在下载文件或者读取显示文件的时候，所要读取或下载的文件，可在参数中直接传递，后台程序获取这个文件路径后，可直接读取返回
问题在于参数使用户可控的，可以直接传入想要读取的文件路径，即可利用。
这种漏洞挖掘起来相对容易，在黑盒测试中，可以先看看功能点对应的文件，再读取文件，这样找起来比较快。在白盒测试中，可以搜索文件读
取的函数，看看有没有直接或者间接可控的变量，常见的文件读取函数有：
file_get_contents(),highlight_file(),fopen(),readfle(),fread(),fgets(),file()等，除了这些正常读取文件的函数外，另外一些其他功能
的函数也一样可以读取文件。比如文件包含函数等，可以利用文件输入输出流php://filter/来读取文件。
关于文件上传：
文件上传漏洞是出现比较早的漏洞，是指把文件上传到管理员或者应用程序不想让你上传的目录中，并不一定是上传一个web服务器可以解析的代
码文件到可以解析的目录。
就目前而言，关于上传漏洞大都是基于黑名单的绕过，不限制文件格式的上传漏洞现在基本上不多了。
而在挖掘该漏洞时，也比较简单。一般的应用可以上传文件的点比较少，其次是现在的web程序大都是利用框架开发的，上传点都是调用同一个类，
而上传函数又就只有move_uploaded_file()这一个。所以文件上传漏洞在白盒测试的时候比较容易挖掘，直接搜索move_upload_file()函数，再看
看调用这个函数上传文件的代码存在不存在上传格式的限制问题，或者是否可以绕过。其中问题比较多的是黑名单限制文件格式以及未更改文件名
而引起的漏洞。在没有更改文件名的情况下，可以利用Apache向前寻找解析格式，IIS6的分号解析漏洞来执行代码。
常见的文件上传漏洞：
（1）未过滤或者本地过滤
它们的共同点都是在服务器端没有过滤，未过滤就是没有限制任何格式的文件上传。本地过滤就是在客户端进行了简单的JS过滤等，这中只要禁用
JS（在不影响其功能的前提下），后者通过BS绕过的方式即可上传任意格式的文件。
（2）黑名单过滤的方式
这种方式因为考虑的扩展名不全存在绕过问题，所以慢慢改为了白名单的方式。毕竟上传的文件格式有着不可预见性，难免有漏网之鱼，例如：IIS
默认解析ASP代码，可它不仅仅解析.asp这个扩展名，还有cdx，asa，cer等，若是在黑名单你没有写全，相当于没有限制。
（3）文件头，content-type验证
在验证文件头时，如果直接上传非图片文件格式，会上传失败，但是若在文件头里面加上 “GIF89a”再上传，就会上传成功。这是因为程序用了一些
不可靠的函数来判断是不是图片文件，比如getimagesize()函数，只要文件头是“GIF89a”就返回正常的图片尺寸数组。
而验证content-type，是在http请求头里面的，这个值可以由请求者任意修改，所以单纯的验证这个值，根本不可靠。
关于文件操作漏洞的通用性防范：
文件操作漏洞有以下几个共同点：
（1）由越权引起来的可操作未授权的文件
（2）要操作更多文件需要跳转目录
（3）大多是在请求中直接传入文件名
所以通用的防御手段：
（1）合理分配权限
（2）避免目录跳转问题，禁止使用’..‘和’\‘。
（3）在操作文件时尽量不直操作文件名，需要传入文件路径时，可以固定文件操作目录。
文件上传漏洞防范：
文件上传漏洞在不同的环境下，利用方式也有所差异。所以需要比较完善的防御策略，因为在修复和防范漏洞之前，要比较清楚的知道这种漏洞在不
同环境下的利用方式，这样才能防御的比较周全。
所以在文件上传的地方要做到以下两点：
（1）采用白名单的过滤方式。
（2）保存上传文件时，要重命名文件。

文件包含

关于文件包含：
PHP的文件包含漏洞可以直接执行包含文件的代码，并且包含文件的格式不受限制，只要能正常执行即可。文件包含分为远程包含和本地包含，
但是两种漏洞都是比较高危的，在渗透与测试过程中该漏洞大都可以直接利用获取Webshell 。文件包含函数有include()，include_once()和
require()，require_once()，其中关于两者的区别，inculde()和include_once()在包含文件遇到错误时，下面代码会继续执行，而require()
和require_once()则会直接报错并退出程序。并且include()是在代码执行到该函数时才将文件包含进来，而require()是在程序开始执行时，
立即包含进来。
漏洞挖掘：
文件包含漏洞大多出现在模块加载，模板加载以及cache调用的地方，例如传入的模块名参数，实际上是直接拼接到文件包含路径中的。所以在
挖掘文件包含漏洞的时候可以先跟踪一下程序运行流程，看看模块加载时的文件是否可控，也可在白盒审计的时候直接搜索include(),require()
直接回溯看看有没有可控变量，一般这类本地包含大多需要截断。
本地文件包含（LFI）：
顾名思义是指只能包含本机文件的文件包含漏洞，大都出现在模块加载，模板加载，cache调用的地方，渗透测试的时候利用起来并不鸡肋，仅仅
需要上传一个允许上传的文件格式再包含该文件来执行代码；也可包含PHP的临时文件，在URL/ua里面加入要执行的代码，当webserver记录到日志后
再包含webserver日志文件；在Linux中也可以包含/proc/self/eenviron文件。
远程文件包含（RFI）：
是指可以包含远程文件的包含漏洞，远程文件包含需要设置allow_url_include=on，在php5.2之后这个选项的可修改范围是PHP_INI_ALL。四个文件包
含函数都支持HTTP,FTP等协议，相当于本地文件包含，它更容易利用，不过出现的频率没有本地包含多。
其中基于HTTP协议的测试代码：
<?php
include($_GET[‘url’])；
?>
只需要在GET请求URL中传入参数即可，（”http://test/test.txt"）；
远程文件包含还有一种输入输出流的利用方式，可以直接执行POST提交的代码。例如：
只要执行POST请求1.php?a=php://input，而在POST的内容里面填写要执行的代码即可。
文件截断：
大多数的文件包含都需要截断，因为正常程序里面包含的文件代码一般像include($test.’.php’)这样的格式，如果我们不能写入以.php为扩展名的文件
那我们是需要截断来利用的。
（1）%00截断虽然很古老，但还是存在。首先，%00受GPC和addslashes等函数的影响，在开启GPC的情况下是不可用的，另外在PHP5.3之后的版本也不可
用，因为已经修复了该漏洞。
（2）另外，也可以利用多个英文句号(.)和反斜杠(/)来接断，这种方式不收GPC影响，但同样在PHP5.3之后不可用。
（3）在远程文件包含中，可利用问号（?）来伪截断，并且不受GPC和PHP版本的影响，其原理是在包含’http://test/test.txt?.php'时，web服务器将？
之后的内容当做参数，txt又不解析，故而参数对test.txt的内容没有影响。所以和包含’http://tets/test.txt'返回的结果一样。从而实现了伪截断。

XSS

关于XSS漏洞：
（1）反射型：通过外部输入直接在在浏览器端触发。
（2）储存型：先把利用代码保存在数据库或文件中，当web程序读取利用代码并输出时触发漏洞。
危害：盗取cookies，修改页面钓鱼等，凡是前端页面能做的，XSS都能做。
XSS出现的地方：
XSS漏洞经常出现在文章发表，评论回复，留言以及资料设置等地方。特别是在文章发表的时候，由于大都是富文本，
各种图片的引用，文字格式的设置等，故而经常出现过滤不严造成的XSS，同样，留言评论也是如此。而在资料设置的
地方，比如用户昵称，签名等，有的应用不止一处设置资料的地方，在注册的地方，修改资料的地方都可以，不一定都
进行了严格的过滤。这几个地方通常都是储存型XSS。
关于反射型：
这种漏洞比较容易，通过黑盒扫描器直接发现，只需要将尖括号，单双引号等直接提交到web服务器，再检查返回的HTML页面
有没有保留原来的特殊字符即可，在白盒测试中，需要寻找带有参数输出的函数，然后根据输出函数对输出内容回溯输入参数，
观察有没有过滤。
关于储存型XSS：
储存型XSS比反射型容易利用的多，不用考虑绕过浏览器的过滤，隐蔽性也比较好，特别在社交网络中，储存型XSS蠕虫病毒易造成
大面积传播，影响巨大。要挖掘储存型XSS也要挖掘未过滤的输入点和未过滤的输出点，然而这个最终的输出点可能跟输入点完全不
在一个业务上。在白盒审计时可以根据当前代码去猜，或者去追查哪里右操作过这些数据，使用表名，字段名在代码里搜索。
漏洞防范：
XSS漏洞在不同的浏览器有不同的利用方式，特别是业务上使用富文本编辑器的时候，防御起来更为复杂。
（1）特殊字符HTML实体转码
需要过滤的字符有单引号，双引号，尖括号，反斜杠，冒号，&，#等。为了保持数据的原始性，最好的的过滤方式是在输出和二次调
用的时候进行HTML实体转码。
（2）标签事件属性黑名单或者白名单
通过过滤特殊字符防止XSS漏洞，实际中过滤了也可能被绕过，比如通过宽字节注入合并掉反斜杠，再利用标签事件来执行js代码，故
而还要通过标签事件的黑白名单的方式，推荐使用白名单的方式。可通过正则匹配的方式，若匹配不在白名单列表直接拦掉。

2018-04-18

CSRF

关于CSRF请求伪造：
就是可以劫持其它用户进行一些请求，而这个CSRF的危害性就取决于该请求是什么操作了，主要用于越权操作故而经常出现在有权限控制的地方。
像管理后台，会员中心，论坛帖子，交易管理等。
利用条件：
（1）浏览器登录了当前所预想的权限用户。
（2）构造实现目标所需要的请求。
挖掘要点：
首先打开几个非静态页面，抓包看看有没有token，如果没有token的话，再直接请求这个页面，若不带referer，并且返回的数据还相同。则很有可
能就是CSRF漏洞了。在白盒测试中，就要看看几个核心文件有没有验证token和referer的相关代码。
漏洞防范：
防御CSRF漏洞的最主要问题就是解决可信问题，即使管理员权限提交到服务器的数据也是不完全可信的，故而做到如下两点防御措施：
（1）增加token、referer验证避免img标签请求的水坑攻击。
（2）增加验证码。
其中token验证是使用最多的一种，也是效果最好的一种，可以理解为在页面或cookie里面加一个不可预测的字符串，服务器在接收操作请求的时候
验证一下这个字符串是不是上次访问留下来的即可判断是不是可信请求，因为不访问上一个页面是无法得到这个token的。除非通过XSS漏洞或者其它
手段得到通信数据。
而考虑到用户体验验证码就没有token实用了，不可能让用户每个页面都输入验证码，说以验证码只用在像登录页面这样的敏感操作中。其实现方式
跟token差不多，都是在上一个页面生成一个随机数，看看前台提交过来的是否一致。
